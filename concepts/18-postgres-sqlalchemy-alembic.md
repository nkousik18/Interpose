# Postgres, SQLAlchemy, and Alembic

## Why a real database now, after everything else has been in-memory or files

Everything Interpose has stored so far has been either ephemeral (the in-memory
`RateLimiter` and `PolicyEngine` from [[16-policy-engine-composition]]) or files
(`config/upstreams.yaml`, policy YAML, the subsampled AML data from
[[14-spark-and-pyspark]]). The audit log is different: it has to outlive the process,
be queryable ("show me every call this agent made last week"), and support
concurrent writers safely. That's what a real database is for, and specifically
**Postgres** â€” chosen in the scoping doc (Section 6.4) over SQLite (fails under
concurrent writers), DynamoDB (expensive, less queryable), and an event-store product
(correct in spirit but an unfamiliar dependency for an MVP).

## SQLAlchemy: what an ORM actually buys you here

**SQLAlchemy** is Python's dominant ORM (Object-Relational Mapper) â€” it lets you
describe a database table as a Python class (`src/interpose/audit/models.py`'s
`AuditEntry`) and get back real Python objects instead of hand-parsed query result
rows. The specific thing it buys Interpose: the *same* `AuditEntry` class is used
to generate the schema (via Alembic, below), to insert rows (`store.py`), and to read
them back for verification (`chain.py`'s caller) â€” one definition of "what an audit
entry looks like," not three copies that can quietly drift out of sync.

Interpose uses SQLAlchemy's newer **async** mode (`create_async_engine`,
`AsyncSession`) for the gateway's own read/write path, because the gateway is
already an async application throughout (FastAPI, httpx) â€” a synchronous database
call would block the whole event loop for every other in-flight request while it
waited on disk I/O. Alembic (below), by contrast, runs synchronously â€” migrations are
a one-shot command-line operation, not something that needs to share an event loop
with anything else.

## Alembic: migrations as a paper trail for schema changes

Hand-editing a database's schema with ad-hoc `ALTER TABLE` statements works until two
people (or two environments â€” dev, staging, prod) drift out of sync with no record of
what changed or when. **Alembic** is SQLAlchemy's migration tool: each schema change
becomes a small, numbered, version-controlled Python file
(`alembic/versions/04ef9f18fb8f_create_audit_entries.py`) with an `upgrade()` and a
`downgrade()`. Running `alembic upgrade head` brings any database â€” a fresh dev
container, CI's ephemeral Postgres, production â€” to the exact same schema state,
starting from wherever it currently is.

The first migration here was **autogenerated**: Alembic compared the empty database
against `AuditEntry`'s SQLAlchemy definition and wrote the `CREATE TABLE` /
`CREATE INDEX` statements needed to reconcile them. That's the normal workflow going
forward too â€” change the model, run
`alembic revision --autogenerate -m "..."`, review what it generated (autogenerate is
a good first draft, not infallible â€” it doesn't always catch things like column
renames), then commit the migration file alongside the model change.

## One source of truth for the connection string

`alembic/env.py` doesn't read a separate database URL from `alembic.ini` â€” it imports
`interpose.config.get_settings()` and uses the same `DATABASE_URL` the running gateway
uses (`src/interpose/config.py`, a `pydantic-settings` `BaseSettings` reading from
the environment / a local `.env` file). One setting, one place it's defined; nothing
to keep in sync between "the app's config" and "the migration tool's config."
